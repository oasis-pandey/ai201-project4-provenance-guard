# Provenance Guard

Provenance Guard is an AI-detection pipeline that evaluates text submissions to determine whether they were likely written by a human or generated by an AI. It processes incoming text through a multi-signal detection pipeline, computes a calibrated confidence score, and returns a transparent classification label.

## Architecture Overview

The path a single submission takes from input to the label a reader sees:

1. **`POST /submit`** receives a JSON body (`text`, `creator_id`) and mints a unique `content_id`.
2. The raw text fans out to the **two detection signals** in the pipeline — Perplexity (LLM) and Burstiness (heuristic) — which run independently and return individual scores.
3. **Confidence scoring** combines the two signal scores into a single calibrated score on a 0.0–1.0 scale (`1.0 - (0.6 * Perplexity + 0.4 * Burstiness)`).
4. **Label routing** maps that score to one of three plain-language transparency labels via fixed thresholds (0.4 and 0.7).
5. A **structured entry** (timestamp, content_id, creator_id, attribution, confidence, both signal scores, status) is written to the SQLite **audit log**.
6. The endpoint returns a JSON response containing the `content_id`, attribution, confidence, both individual signal scores, and the transparency label text.

If a creator disputes a result, **`POST /appeal`** takes the `content_id` plus `creator_reasoning`, flips the logged status to `under_review`, records the reasoning alongside the original classification in the audit log, and returns a confirmation. **`GET /log`** exposes the most recent audit entries as JSON for inspection.

### API Surface

| Endpoint | Method | Accepts | Returns |
|----------|--------|---------|---------|
| `/submit` | POST | `text`, `creator_id` | `content_id`, `attribution`, `confidence`, `llm_score`, `burstiness_score`, `label` |
| `/appeal` | POST | `content_id`, `creator_reasoning` | confirmation + `status: under_review` |
| `/log` | GET | — | most recent audit-log entries |

## Architecture and Design Decisions

The application exposes a REST API built with Flask, backed by a SQLite database for audit logging. The core architecture relies on a **two-signal detection pipeline**:

1. **Signal 1: Perplexity (LLM-based)**
   - **Reasoning**: AI models predict the next most likely token. As a result, AI-generated text tends to follow highly predictable patterns with low "surprise" (low perplexity). Human writing is inherently more unpredictable and creative. We use the Groq API (Llama 3) to evaluate the text's perplexity and return a normalized predictability score.
2. **Signal 2: Burstiness (Heuristic-based)**
   - **Reasoning**: Humans naturally vary their sentence lengths—mixing short, punchy statements with longer, complex ones. AI text tends to fall into a uniform rhythm. This signal splits the text into sentences and calculates the standard deviation of their word counts.

**Why this approach?**
Relying solely on LLM-based perplexity is fragile; it often flags formal human writing (which is naturally predictable) as AI. By combining perplexity with a structural heuristic (burstiness), the system cross-references word-choice predictability against sentence-structure variation, creating a much more robust classifier.

If deploying this in a real-world scenario, I would replace the naive standard-deviation heuristic for burstiness with a more sophisticated NLP parser (like spaCy) to analyze syntax tree depth, and I would swap the LLM-prompted perplexity for a dedicated, smaller model fine-tuned specifically for classification to reduce latency and API costs.

## Confidence Scoring & Calibration

The system combines both signals using a weighted average: `1.0 - (0.6 * Perplexity + 0.4 * Burstiness)`. 
The raw burstiness standard deviation is min-max normalized between 5.0 and 10.0 to heavily penalize uniformity.

**What "confidence" means here:** the combined score is a 0.0–1.0 estimate of how AI-like the text is. The system is *most confident* when a score sits near the extremes (close to 0.0 = clearly human, close to 1.0 = clearly AI) and *least confident* when a score lands in the middle of the range, which is exactly why the wide `0.4–0.7` "Uncertain" band exists. A 0.95 and a 0.55 are not just different numbers — they trigger different labels, because the system should not make a firm accusation when its own signals disagree. This also reflects a deliberate asymmetry: on a writing platform, falsely labeling a human's work as AI is worse than missing an AI, so borderline scores are routed to "Uncertain" rather than "AI."

### Validation: testing inputs that should produce different scores

To verify the scoring is meaningful rather than constant, we ran four deliberately chosen inputs spanning the range — two clear cases and two borderline cases — and confirmed the scores land where intuition predicts. These are real, reproducible outputs from the pipeline:

| Input type | Perplexity | Burstiness | Combined score | Verdict |
|------------|-----------|-----------|----------------|---------|
| Clearly AI (uniform, formulaic essay) | 0.42 | 5.44 | **0.713** | AI |
| Clearly human (casual, irregular review) | 0.92 | 6.72 | **0.310** | Human |
| Borderline: formal human (academic prose) | 0.62 | 5.50 | **0.588** | Uncertain |
| Borderline: lightly edited AI (mixed) | 0.72 | 4.97 | **0.568** | Uncertain |

The clear cases separate cleanly (0.310 vs 0.713), and — importantly — the two borderline inputs do **not** get forced into a confident verdict; they correctly fall into the Uncertain band.

### Example Submissions: high-confidence vs. lower-confidence

**High-Confidence case — Score: 0.310 (Human)**
*Input:* "ok so i finally tried that new ramen place downtown and honestly? underwhelming. the broth was fine but they put WAY too much sodium in it and i was thirsty for like three hours after."
*Result:* The high structural variation and unpredictable, casual phrasing yield a low AI-score of **0.310** — far from the 0.5 midpoint, so the system is **confident** this is human and returns the firm "Verified Human-Written" label.

**Lower-Confidence case — Score: 0.568 (Uncertain)**
*Input:* "I've been thinking a lot about remote work lately. There are genuine tradeoffs — flexibility and no commute on one side, isolation and blurred work-life boundaries on the other. Studies show productivity varies widely by individual and role type."
*Result:* This lightly-edited-AI text mixes some predictability with moderate structural variation, landing at **0.568** — close to the midpoint. The system is **not confident** either way, so it returns the hedged "Uncertain/Mixed" label rather than accusing the creator.

## Transparency Label Variants

Based on the final confidence score (0.0 to 1.0), the system returns one of three transparent labels:

1. **High-Confidence Human** (Score 0.0 to 0.4):
   > "Verified Human-Written: The structural variation and word choices in this text strongly align with human writing patterns."
2. **Uncertain** (Score 0.4 to 0.7):
   > "Uncertain/Mixed: This text exhibits a combination of predictable patterns and style variations. It may be human work that is highly structured, or AI-generated text that has been edited."
3. **High-Confidence AI** (Score 0.7 to 1.0):
   > "Likely AI-Generated: This text contains highly predictable word choices and uniform sentence structures characteristic of AI generation."

## Known Limitations

**Structured Poetry and Art Pieces**
The system relies heavily on the "Burstiness" signal (variance in sentence length) to identify human text. Human-written rhythmic poetry, sonnets, or heavily stylized art pieces intentionally utilize uniform sentence structures, consistent lengths, and repetitive word selections. Because this style deliberately lacks structural variance, the Burstiness signal will register low variation, causing the system to misclassify this highly creative human-written work as AI-generated.

## Spec Reflection

- **How the spec helped:** The spec provided a strict scale for the confidence mapping (0.0 to 1.0) and rigid threshold boundaries (0.4 and 0.7). This forced me to build a dedicated calibration script to tune the Burstiness normalization until the scores mathematically landed exactly in the designated zones, preventing arbitrary "gut-feeling" score assignments.
- **How the implementation diverged:** The original spec assumed the app would run on port 5000. During testing, `curl` requests were failing with empty responses. I discovered that on macOS Monterey and later, Apple's AirPlay Receiver monopolizes port 5000. I diverged from the spec by moving the Flask server to port 5001 to ensure the API was actually reachable on local development machines.

## AI Usage

1. **Rate Limiting Setup**
   - **Direction**: I directed the AI to set up Flask-Limiter on the `/submit` endpoint using an in-memory storage URI.
   - **Produced**: The AI correctly imported `Limiter` and generated the `@limiter.limit("10 per minute;100 per day")` decorator.
   - **Revision**: No heavy logic revision was needed, but we tested the AI's implementation manually with a bash loop to ensure the `429` status codes successfully fired.

2. **Burstiness Normalization**
   - **Direction**: I directed the AI to write a standalone `get_burstiness_score` function that calculates the standard deviation of sentence lengths.
   - **Produced**: The AI generated the function but initially used a flat cap of 15 words for the max normalization boundary.
   - **Revision**: After testing the raw outputs, I realized a flat cap of 15 compressed the scores too much. I overrode the AI's initial math and rewrote the normalization bounds to explicitly use a `min=5.0` and `max=10.0` range, which perfectly calibrated the scores to match the spec's thresholds.

## Rate Limiting & Appeals Evidence

### Rate Limiting Policy & Reasoning

The `/submit` endpoint enforces **10 requests per minute** and **100 requests per day**, per client IP (configured via `@limiter.limit("10 per minute;100 per day")`).

These limits are tied to how the endpoint is actually used on a writing platform rather than to a library default:

- **10 / minute** — A real creator checks a draft, edits a sentence or two, and re-checks. That iterative loop produces a small burst of submissions in a short window, so the per-minute ceiling is set high enough to never interrupt legitimate editing, but low enough to block scripted abuse. The most likely abuse here is someone scripting the endpoint to use our perplexity call as a free "AI-detection oracle" or to scrape model behavior; 10/min makes that economically pointless without affecting a human at the keyboard.
- **100 / day** — Even a prolific writer rarely runs more than a few dozen distinct pieces through a checker in a single day. A daily cap of 100 sits comfortably above genuine creator usage while protecting the system from two real costs: the paid Groq API bill (every submission is a billed LLM call, so an unbounded client could run up cost quickly) and sustained low-and-slow scraping that would slip under the per-minute limit. The day-level cap is the backstop that the minute-level limit can't catch.

A hard 429 (rather than a silent queue) is returned so the client gets immediate, honest feedback that they've hit the limit.
